## 摘要

当下主流预测市场集中在 **Kalshi** 与 **Polymarket** 两个平台。本文针对主流预测市场平台 Kalshi 与 Polymarket 在撮合、更新与结算机制上的结构差异，介绍了：

1. **双边对冲套利** — 同标的跨平台价差未同步时，双边持仓锁定收益
2. **Lag 套利** — 以Kalshi为价格锚，在Polymarket延迟窗口内限价捕捉

---

## 一、平台机制对比

### 1.1 Kalshi — 传统金融监管

| 特点 | 说明 |
|------|------|
| 监管 | CFTC 监管 (DCM指定合约市场) |
| 资金 | 美元银行体系 (ACH/Wire) |
| 产品 | 事件合约 (Yes/No)，价格 0-100 美分 |
| 撮合 | 传统交易所方式，标准化 |

**融资**: E轮 $10亿，估值 $110亿 (a16z, Sequoia, Paradigm, Google CapitalG, Coinbase Ventures等)

### 1.2 Polymarket — 去中心化链上

| 特点 | 说明 |
|------|------|
| 架构 | 链下撮合、链上结算 (CLOB) |
| 资金 | 稳定币 (USDC等) |
| 产品 | 预测合约 |
| 结算 | 链上合约 |

**融资**: 与纽交所母公司ICE谈判 $20亿投资 (Founders Fund, Vitalik Buterin, Dragonfly Capital等)

## 二、套利机制

### 2.1 双边对冲套利

当同一标的在两个平台价格短期不同步时：

1. 在低价平台买入
2. 在高价平台卖出
3. 价差收敛或到期结算时锁定收益

### 2.2 Lag 套利

Polymarket 采用链下撮合 + 链上结算结构，价格更新存在延迟：

1. 以 **Kalshi** 作为价格锚（调整更快）
2. 在 **Polymarket** 侧挂限价单
3. 捕捉尚未更新的报价

---

## 三、Kalshi 机制特点

### 3.1 订单结构

- Yes/No 互补结构：YES + NO = 100美分
- 盘口只显示 YES 买单和 NO 买单
- 需要通过互补关系计算实际卖出价格

**注意**: 必须在同一语义下比较盘口，否则会把读错盘口当成价差

### 3.2 API 限速

| 账户等级 | 读请求 | 写请求 |
|----------|--------|--------|
| Basic | ~20/秒 | ~10/秒 |
| Advanced | 30/秒 | 30/秒 |
| Premier | 100/秒 | 100/秒 |
| Prime | 数百/秒 | 数百/秒 |

**常见坑**:
1. **撤单也算写请求** — 容易打满限速
2. **重试会放大流量** — 必须在策略层统一调度
3. **签名时间偏差** — 需要 NTP 校准

### 3.3 资金路径

```
USDC → 交易所 → USD → 银行 → Kalshi
```

- ACH 入金: 1-3天
- Wire 出金: T+1
- 大额可能触发人工审核
- 整体: 2-4个工作日

**资金占用**: 双边对冲需要双倍资金 (20k市场规模需要~40k资金)

## 四、Polymarket 机制特点

### 4.1 订单结构

- EIP712 标准签名消息
- 链下撮合 → 链上结算

**生命周期**:
1. 链下撮合确认 (成交回执)
2. 链上结算确认 (资产转移)

### 4.2 API 分层

| API | 职责 |
|-----|------|
| CLOB API | 交易与盘口 |
| Gamma API | 市场元数据 |
| Data API | 仓位与余额 |
| WebSocket | 实时推送 |

### 4.3 限速特点

- Cloudflare 10秒滑窗限速
- 短时间集中爆发易触发限流
- 触发表现: HTTP 429 / 请求延迟 / 被拦截

### 4.4 订单类型

- **post-only**: 只挂盘口，不主动吃对手价
- 对 Lag 套利非常关键

## 五、实操建议

### 5.1 资金管理

1. 两平台都预留静态库存
2. 不要依赖实时资金调度
3. 不要用刚入金的资金立刻做大规模对冲
4. 5个市场同时跑可能需要 $200k+ 资金

### 5.2 API 策略

1. 读请求预算控制在 50% 上限
2. 写请求预算控制在 50% 上限
3. 预留 30% 作为缓冲
4. **撤单和平仓永远优先于开仓**

### 5.3 风险控制

- 部分成交 → 单边敞口暴露
- 撤单延迟 → 旧单被反向吃掉
- 风控限制 → 仓位冻结

## 六、关键结论

| 维度 | Kalshi | Polymarket |
|------|---------|-------------|
| 价格更新 | 快 (传统交易所) | 慢 (链上延迟) |
| 资金效率 | 低 (美元体系慢) | 高 (稳定币实时) |
| API 限速 | 明确分层 | Cloudflare 滑窗 |
| 套利方式 | 价格锚 | 延迟捕捉 |

**核心**: 以 Kalshi 为价格参照，在 Polymarket 捕捉延迟窗口
